## Ключевые Термины

**Предмет** — объект полученный со сцены или при входе в игру. Не участвует в личных/коллективных целях пока находится у игрока. При первом дарении НЕ меняет вид, но становится подарком для получателя.

**Подарок** — объект полученный от другого игрока (либо трансформированный предмет при первом дарении, либо передаренный подарок). Участвует в личной коллекции и коллективной цели. При передаривании трансформируется в другой подарок детерминистически по ID.

## 5.1: Systems Specification

### Система 1: Сцена и Сбор Предметов

**Что делает игрок:**
- Открывает экран сцены (большая статическая иллюстрация праздничного пространства)
- Pan/zoom по сцене, ищет предметы визуально
- Видит предмет (иконка/sprite наложен на сцену) → тапает → собирает в инвентарь

**Триггеры:**
- Игрок открывает экран сцены
- Игрок тапает на видимый предмет

**Результат:**
- Предмет исчезает со сцены для ВСЕХ игроков (глобальный пул)
- Предмет появляется в инвентаре собравшего игрока

**Правила:**
- Глобальный пул: если один собрал → исчезает для всех
- Респавн: каждые 5-10 минут (точнее на балансировке) добавляется 3-5 новых предметов
- Размещение: 8-12 фиксированных точек на сцене, новые предметы появляются на свободных точках
- First-come-first-served: кто быстрее тапнул, тот получил
- Предметы визуально заметны (красивые иконки для Wonder aesthetic)

---

### Система 1б: Периодические Предметы

**Что происходит:**
- Раз в T минут (параметр для L7) каждый авторизованный игрок получает 1 предмет из каталога
- Предмет добавляется напрямую в инвентарь, без взаимодействия с игроком
- Предметы случайные (случайный id из каталога)

**Триггеры:**
- Серверный таймер: каждые T минут, для каждого зарегистрированного пользователя

**Результат:**
- Предмет появляется в инвентаре игрока
- Если игрок онлайн: badge инвентаря обновляется (или тихо появляется при следующем открытии)

**Правила:**
- Выдаётся независимо от того, онлайн игрок или нет (придёт при следующем заходе)
- Нет лимита на накопление — предметы копятся в инвентаре
- Мотивация: у любого игрока всегда есть что подарить, даже если он не заходил на сцену

---

### Система 2: Инвентарь

**Что делает игрок:**
- Открывает инвентарь → видит всё что у него есть (предметы + подарки)
- Может тапнуть на предмет/подарок → открывается детальный просмотр:
  - Картинка побольше
  - Описание
  - **Если подарок:** от кого (если не анонимно), когда был подарен
  - **Если предмет:** когда собран со сцены
- Может выбрать что подарить (переход к Системе 3: Дарение)

**Триггеры:**
- Игрок открывает экран инвентаря
- Игрок тапает на конкретный предмет/подарок (открывает детальный просмотр)
- Предмет собран со сцены → добавляется в инвентарь
- Подарок получен от другого игрока → добавляется в инвентарь

**Результат:**
- Игрок видит весь список предметов/подарков доступных для дарения
- При тапе — видит детали конкретного предмета/подарка (картинка, описание, история)

**Правила:**
- В инвентаре хранятся И предметы И подарки
- Предметы ≠ Подарки (разная механика трансформации при дарении)
- Нет искусственного лимита на размер инвентаря
- Каждый предмет/подарок имеет уникальный ID (для детерминистической трансформации)
- История хранится: кто подарил (если не анонимно), когда (timestamp)

---

### Система 3: Дарение и Трансформация

**Что делает игрок:**
1. Тапает на предмет/подарок в инвентаре → открывается детальный просмотр
2. Кнопка "Подарить это" → переход к экрану дарения
3. Выбирает получателя из списка коллег
4. **Если дарит подарок:** видит превью трансформации — "Ты даришь [X], получатель получит [Y]"
5. **Если дарит предмет:** превью показывает "Получатель получит подарок [X]"
6. **Опционально:** добавляет сообщение (текстовое поле)
7. Выбирает анонимность (галочка/toggle)
8. Подтверждает дарение

**Триггеры:**
- Игрок выбирает "Подарить это"
- Игрок подтверждает дарение

**Результат:**
- Предмет/подарок исчезает из инвентаря дарителя
- Получатель получает подарок (трансформированный или нет, в зависимости от типа)
- Если было сообщение: получатель видит его при раскрытии подарка
- В БД записывается: кто → кому, что, когда, анонимно/открыто, текст сообщения (если есть)
- Коллективная цель увеличивается на +1 подарок
- Получателю приходит уведомление "Новый подарок!"

**Правила:**

**Трансформация при дарении:**
- **Предмет → Подарок (первое дарение):** НЕ меняет визуальный вид, но меняет статус (становится подарком для получателя, участвует в его коллекции и коллективной цели)
- **Подарок → Подарок (передаривание):** Трансформируется детерминистически по уникальному ID экземпляра подарка
  - Один и тот же экземпляр (по ID) всегда трансформируется в одно и то же
  - Показывается превью ДО подтверждения дарения
  - Игрок может передумать и выбрать другой предмет/подарок

**Анонимность:**
- Решение каждый раз при дарении (не "всегда анонимно")
- Если анонимно: получатель видит "Тебе подарили [X]!" (без имени дарителя)
- Если открыто: получатель видит "[Имя] подарил тебе [X]!"

**Сообщение:**
- Опциональное — можно оставить пустым
- Доступно и в анонимных подарках — получатель видит сообщение, но не знает от кого (если анонимно)
- При раскрытии подарка: сообщение показывается вместе с подарком

**Выбор получателя:**
- В списке получателей доступны ВСЕ пользователи которые хоть раз авторизовались в игре (не только онлайн сейчас)
- Можно подарить коллеге который офлайн — подарок придёт ему при следующем заходе
- Своё имя скрыто из списка — нельзя подарить себе

---

### Система 4: Получение Подарков

**Что делает игрок:**
1. Получает уведомление "У тебя X новых подарков!" при заходе в игру
2. Открывает раздел с нераскрытыми подарками (wrap/упакованные подарки)
3. Тапает на упакованный подарок → анимация раскрытия (критично для Wonder + Gift-Giving Delight)
4. Видит результат:
   - Картинка подарка (большая)
   - Описание
   - От кого (если не анонимно) / "Анонимный подарок" (если анонимно)
   - Сообщение от дарителя (если есть)
   - Когда подарен
5. Подарок автоматически добавляется в инвентарь после раскрытия

**Триггеры:**
- Другой игрок подарил тебе предмет/подарок → создаётся нераскрытый подарок
- Игрок тапает на нераскрытый подарок → начинается раскрытие

**Результат:**
- Нераскрытый подарок исчезает из списка "новые"
- Подарок добавляется в инвентарь (уже раскрытый)
- Игрок переживает момент сюрприза

**Правила:**
- До раскрытия содержимое скрыто
- Раскрытие обязательно для доступа к содержимому (нельзя "автоматически добавить всё в инвентарь")
- Нераскрытые подарки персистентны — не исчезают если игрок вышел
- Можно раскрывать в любом порядке

---

### Система 5: Коллективная Цель

**Что делает игрок:**
- Наблюдает визуальное пространство (возможно, тот же экран что сцена для сбора предметов, или отдельный экран "Наш праздник")
- Видит как пространство постепенно заполняется/украшается с каждым подарком
- В 19:00 — видит финальное празднование достижения цели (кульминация)

**Триггеры:**
- Любой игрок подарил предмет/подарок → коллективный прогресс +1
- 12:00, 15:00, 17:00 — автоматический пересчёт цели (серверная логика)
- 19:00 — финал

**Результат:**
- Визуальное пространство наполняется — добавляются элементы/украшения/подарки
- При достижении 100% — визуальная кульминация (праздничная анимация, завершённая картина)
- В 19:00 — финальное событие для всех игроков (shared triumph)

**Правила:**

**Адаптивная логика (скрыта от игроков):**
- Система считает суммарное количество подарков которые УЖЕ подарены друг другу
- **Пересчёт цели:** В 12:00, 15:00, 17:00:
  1. Смотрим сколько подарков подарено к этому моменту
  2. Экстраполируем темп: "К 19:00 будет примерно X подарков"
  3. Устанавливаем цель = X × 0.9 (достижимо, но требует усилия)
  4. Между пересчётами цель фиксирована
- **Bounds:** Цель не может быть меньше 300 или больше 1500 подарков
- **Визуализация:** Прогресс показывается как % наполненности пространства, БЕЗ точных цифр ("347/800")

**Визуализация прогресса:**
- Пространство постепенно заполняется визуальными элементами (подарки, украшения)
- Игроки видят "много/мало осталось" по визуальной плотности, а не по цифрам
- Возможно лёгкий текстовый намёк: "Почти готово!" / "Ещё чуть-чуть!" (без точных цифр)

**Финал (19:00):**
- Если цель достигнута: Праздничная кульминация, визуальное празднование
- Если цель НЕ достигнута (маловероятно): Всё равно празднуем усилие, позитивный финал
- Игра завершается, можно вернуться посмотреть результат позже (опционально)

**Edge cases:**
- Если цель достигнута раньше 19:00: Прогресс = 100%, сообщение "Ура! Финал в 19:00!", можно продолжать дарить (прогресс не переполняется)
- Если никто не играл: Bounds гарантируют минимум 300 подарков как цель (достижимо даже при низкой активности)

### Система 6: Личная Коллекция

**Что делает игрок:**
- При заходе видит тройку целевых подарков: "Собери: Мишка, Цветы, Миньон"
- Получает подарки от коллег → проверяет, входят ли в тройку целей
- Если подарок из тройки → автоматически засчитывается ✓
- Когда собрал все три → тройка закрывается, генерируется новая тройка
- Может просматривать историю закрытых троек (опционально)

**Триггеры:**
- Первый заход в игру → генерируется первая тройка целей
- Получил подарок от коллеги (Система 4) → проверка "входит ли в текущую тройку?"
- Закрыл все три цели → генерируется новая тройка

**Результат:**
- Цель из тройки отмечается как выполненная ✓
- При закрытии тройки: визуальное празднование (микро-победа), новая тройка генерируется
- Подарок остаётся в инвентаре (можешь передарить дальше или оставить себе)

**Правила:**

**Что засчитывается в цели:**
- ТОЛЬКО подарки полученные от других игроков (через Систему 4: Получение подарков)
- Предметы собранные со сцены НЕ засчитываются (даже если название совпадает с целью)
- Логика: цель мотивирует социальное взаимодействие — просить коллег подарить нужное

**Генерация тройки:**
- Три случайных подарка из каталога (40-60 подарков)
- Могут повторяться между тройками (в разные закрытия тройки может выпасть "Мишка" снова)

**После засчитывания:**
- Подарок остаётся в инвентаре — можешь передарить дальше (щедрость не конфликтует с личной целью)
- Или оставить себе в инвентаре (память о подарке)

**Обновление тройки:**
- Новая тройка генерируется ТОЛЬКО когда закрыл все три цели текущей тройки
- Если зашёл → вышел → зашёл снова: тройка та же самая (нет потери прогресса)

**Связь с коллективной целью:**
- Личная коллекция НЕ конфликтует с коллективной целью — оба счётчика растут параллельно
- Каждый подарок полученный игроком участвует И в его личной тройке целей (если подходит) И в коллективной цели

### Система 7: Авторизация и Профиль

**Что делает игрок:**

1. **Первый заход (регистрация):**
   - Открывает игру → экран "Введи имя и пароль"
   - Вводит имя + придумывает пароль → подтверждает
   - Создаётся аккаунт, попадает в игру

2. **Последующие заходы на том же устройстве:**
   - Автоматический логин (сохранён токен в localStorage/cookie)
   - Сразу попадает в игру

3. **Заход с другого устройства:**
   - Экран "Войти" → вводит имя + пароль → подтверждает
   - Попадает в игру с тем же аккаунтом (инвентарь, коллекция сохранены)

4. **Изменение имени:**
   - Настройки → "Изменить имя" → вводит новое имя → подтверждает
   - Имя обновляется везде (список получателей, история дарений)

5. **Изменение пароля:**
   - Настройки → "Изменить пароль" → вводит старый + новый → подтверждает

**Триггеры:**
- Нет токена → экран регистрации/логина
- Есть токен → автологин
- Игрок открывает настройки → доступ к изменению имени/пароля

**Результат:**
- Создаётся запись: {user_id, имя, хэш_пароля, timestamp_регистрации, timestamp_последнего_захода}
- При логине: создаётся токен, сохраняется на клиенте
- Изменение имени/пароля → обновление в БД

**Правила:**
- Имя обязательно, пароль обязательный (минимум N символов — открытый вопрос для L6)
- Имена могут совпадать у разных игроков (user_id уникален, имя — нет)
- Токен истекает через X времени (или бессрочный для casual формата — открытый вопрос)
- Пароль хранится в виде хэша (bcrypt/argon2)

## 5.2: Integration & Conflicts

### Data Flow

Схема движения предметов и подарков между системами:

```
Сцена (предметы респавнятся)
    ↓ (игрок собирает)
    ├─────────────────────────────────┐
Периодическая выдача (сервер, раз в T мин)
    ↓ (автоматически)
Инвентарь [ПРЕДМЕТ]
    ↓ (игрок дарит)
Система Дарения → трансформация → [ПОДАРОК]
    ↓
Система Получения (у другого игрока)
    ↓ (раскрывает)
Инвентарь [ПОДАРОК]
    ↓
    ├─→ (передаривание) → Система Дарения → трансформация → [ПОДАРОК'] → ...
    ├─→ (проверка) → Личная Коллекция (засчитывается если в тройке целей)
    └─→ (любое дарение) → Коллективная Цель (+1)
```

**Ключевые узлы:**
- Предметы появляются со сцены (сбор) ИЛИ через периодическую выдачу (автоматически)
- Подарки появляются ТОЛЬКО через дарение (первое или передаривание)
- Коллективная цель растёт при ЛЮБОМ дарении (предмет→подарок или подарок→подарок)
- Личная коллекция засчитывает ТОЛЬКО полученные подарки (не предметы)

---

### Conflicts Matrix

Потенциальные конфликты между системами и их резолюция:

| Системы в конфликте | Что может пойти не так | Приоритет / Резолюция |
|---------------------|------------------------|------------------------|
| **Сцена + Инвентарь** | Два игрока одновременно тапнули один предмет со сцены | **Серверная race condition**: Кто первый отправил запрос — тот получил. Второму: ошибка "Уже собран" (вопрос для L6: как показать?) |
| **Дарение + Получение (офлайн)** | Игрок A дарит подарок игроку B, но B офлайн | **Не конфликт**: Подарок сохраняется в БД для B, придёт ему при следующем заходе (система асинхронная) |
| **Дарение + Получение (онлайн)** | Игрок A дарит подарок игроку B, B онлайн прямо сейчас | **Вопрос для L6**: Когда B видит уведомление? (сразу всплывающее / при обновлении экрана / при след. заходе) |
| **Инвентарь + Личная коллекция** | Подарок засчитался в тройку целей, но игрок передарил его дальше | **Не конфликт**: Цель уже засчиталась при получении. Передаривание не сбрасывает прогресс коллекции. |
| **Коллективная цель + Личная коллекция** | Один подарок обновляет оба счётчика одновременно | **Не конфликт**: Это фича — каждый полученный подарок участвует в обоих (если подходит под тройку). Два независимых обновления. |
| **Дарение (трансформация) + Детерминизм** | Один и тот же подарок (по ID) должен всегда трансформироваться в один и тот же результат | **Резолюция**: Lookup table — при создании подарка сразу генерируем "во что он превратится". Храним в БД: {id: 12345, transforms_to: 678}. Можно гарантировать что "Мишка" не превратится в "Мишку". |
| **Получение + Раскрытие** | Игрок получил подарок, но не раскрыл → накопилось 10+ нераскрытых | **Не конфликт**: Нераскрытые подарки персистентны, нет лимита. Игрок может раскрыть все сразу или постепенно. |
| **Респавн предметов + Переполнение сцены** | Предметы респавнятся каждые 5-10 минут, но никто не собирает → все 12 точек заняты | **Резолюция**: Новые предметы НЕ добавляются если все точки заняты. Респавн срабатывает только на свободных точках. |
| **Авторизация + Дублирующиеся имена** | "Александр" зарегистрировался дважды с разных устройств | **Не конфликт**: Имена могут совпадать, user_id уникален. В списке получателей оба "Александра" видны. Игроки различают вживую. |
| **Дарение + Дарение себе** | Игрок пытается выбрать своё имя в списке получателей | **Резолюция**: Своё имя скрыто из списка — технически невозможно подарить себе. |

---

### State Persistence

Что где хранится:

**Server State (БД):**

**Глобальное:**
- **Каталог предметов/подарков:** {id, тип (предмет/подарок), название, изображение, описание, transforms_to (предгенерировано для подарков)}
- **Предметы на сцене:** {id, ссылка_на_каталог, позиция_на_сцене, timestamp_респавна}
- **История всех дарений:** {даритель_id, получатель_id, предмет/подарок_id, анонимно (bool), сообщение (text), timestamp}
- **Коллективная цель:** {текущий_счёт_подарков, целевое_значение, timestamp_последнего_пересчёта}
- **Список всех пользователей:** {user_id, имя, хэш_пароля, timestamp_регистрации, timestamp_последнего_захода}

**Персональное (на пользователя):**
- **Инвентарь:** [{экземпляр_id, ссылка_на_каталог, тип (предмет/подарок), timestamp_получения}]
- **Нераскрытые подарки:** [{экземпляр_id, ссылка_на_каталог, ссылка_на_запись_в_истории_дарений, timestamp_дарения}]
- **Личная коллекция:** {текущая_тройка_целей: [каталог_id1, каталог_id2, каталог_id3], прогресс: [bool, bool, bool], закрытых_троек_всего: N}

**Client State (локально у игрока, синхронизируется):**
- Видимые предметы на сцене (кэш от сервера)
- Свой инвентарь (кэш от сервера)
- Свои нераскрытые подарки (кэш от сервера)
- Своя тройка целей (кэш от сервера)
- Визуальный прогресс коллективной цели (кэш от сервера)

**Синхронизация:**
- При заходе в игру: клиент запрашивает свежее состояние от сервера
- При действиях (сбор, дарение, раскрытие): клиент отправляет запрос → сервер обновляет состояние → возвращает новое состояние клиенту
- При получении подарка от другого игрока (если онлайн): сервер пушит уведомление клиенту (WebSocket / polling — открытый вопрос для реализации)

**Логика:**
- **Каталог** — один для всех, статический (40-60 записей)
- **Экземпляры** в инвентарях — ссылаются на каталог, но имеют уникальные ID (чтобы отличить "мой Мишка" от "твоего Мишки")
- **История дарений** — глобальная таблица, можно выбрать "подарки полученные user_X" или "подарки подаренные user_Y"

### Edge Cases

Нетипичные ситуации и как система их обрабатывает:

| Ситуация | Что происходит | Как система реагирует |
|----------|----------------|----------------------|
| **Первый игрок в день** | Зашёл в 10:00, никого больше нет | Видит пустую сцену → респавн предметов начинается → может собирать и дарить, но получать не от кого. Коллективная цель инициализируется при первом дарении. |
| **Игрок офлайн весь день** | Зарегистрировался, но не зашёл больше | Остаётся в списке получателей (другие могут дарить ему). Подарки накапливаются. Придут при следующем заходе (даже на следующий день/позже). |
| **Очень низкая активность** | 5-10 игроков вместо 20-30 | Адаптация коллективной цели: при пересчёте (12:00, 15:00, 17:00) цель снижается до минимума 300 подарков. Финал в 19:00 всё равно празднуется. |
| **Очень высокая активность** | 40-50 игроков вместо 20-30 | Адаптация коллективной цели: цель повышается до максимума 1500 подарков. Респавн предметов может не успевать → конкуренция за предметы на сцене (это ок, мотивирует заходить чаще). |
| **Цель достигнута рано** | К 16:00 уже 100% коллективной цели | Прогресс = 100%, появляется сообщение "Ура! Финал в 19:00!". Можно продолжать дарить (прогресс не переполняется, визуально остаётся на 100%). Финал всё равно в 19:00. |
| **Респавн когда все точки заняты** | 12 точек заняты, пришло время респавна | Новые предметы НЕ добавляются. Ждут пока освободится хотя бы одна точка. Респавн сработает для свободных точек при следующей проверке. |
| **Дублирующиеся имена** | "Александр" зарегистрировался дважды с разных устройств | Разрешено — имена могут совпадать, user_id уникален. В списке получателей оба "Александра" видны. Можно дарить любому. Игроки различают вживую ("Александр из отдела X"). |
| **Забыл пароль** | Игрок не может войти | Обращается к тебе → ты вручную сбрасываешь пароль в БД (30 человек, не критично). Или открытый вопрос: автоматизировать через email? (нет времени) |
| **Нет подарков из тройки целей** | Игрок получил 10 подарков, но ни один не из его тройки | Ничего не засчитывается в личную коллекцию. Игрок может передарить эти подарки дальше (помогает другим). Тройка остаётся открытой до получения нужных подарков. |
| **Переполнение нераскрытых подарков** | Игрок получил 20 подарков, но не раскрывает их | Не проблема — нет лимита на нераскрытые подарки. Они персистентны. Игрок может раскрыть когда хочет. Возможно напоминание в UI (badge "20!"). |
| **Финал в 19:00, игрок офлайн** | Игрок не зашёл к 19:00 | Финал происходит для всех кто онлайн. Офлайн игрок может посмотреть результат позже (если реализуем экран "Итоги дня"). |
| **Игрок дарит сам себе** | Пытается выбрать своё имя в списке получателей | **Запрещено:** Своё имя скрыто из списка получателей, нельзя выбрать. Можно дарить только другим игрокам. |

## 5.3: Outputs for Level 6 & 7

### Для Level 6 (Presentation Architecture)

#### UI Экраны/Компоненты

**Основные экраны:**
1. **Авторизация/Регистрация** — ввод имени + пароля (первый заход или вход)

7. **Нераскрытые подарки** — список wrap/упакованных подарков
8. **Раскрытие подарка** — анимация, результат (картинка, описание, от кого, сообщение)
9. **Коллективная цель** — визуализация прогресса (заполнение пространства)
10. **Личная коллекция** — тройка целей с прогрессом
11. **Настройки/Профиль** — изменение имени, изменение пароля
12. **Финал (19:00)** — кульминационное событие, визуальное празднование

#### Открытые Вопросы для Level 6

**Из Системы 1 (Сцена):**
- Как отображать race condition если два игрока почти одновременно собрали предмет? (мгновенное исчезновение / fade-out / shake + "Уже собран")
- Как показывать пропажу предмета для игрока который смотрит на сцену когда другой собрал? (real-time WebSocket / polling / обновление при refresh)

**Из Системы 2 (Инвентарь):**
- Визуальное различие подтверждено: Предметы = серая рамка, Подарки = золотая рамка + gift icon

**Из Системы 3 (Дарение):**
- Лимит длины сообщения от дарителя? (рекомендую 100-150 символов)
- Сортировка списка получателей? (алфавит / последние активные / случайная?)
- Показывать ли статус онлайн/офлайн рядом с именем получателя?

**Из Системы 4 (Получение):**
- Что игрок видит на нераскрытом подарке ДО раскрытия? (полностью скрыто / силуэт / категория / размер упаковки?)
- Как показывать получение подарка если игрок онлайн в момент дарения? (всплывающее сразу / badge / обновление при refresh / только при перезаходе)

**Из Системы 5 (Коллективная цель):**
- Как визуальное пространство для коллективной цели соотносится со сценой сбора предметов? (один экран с двумя слоями / отдельные экраны / разные виды одного пространства?)

**Из Системы 6 (Личная коллекция):**
- Как визуализируется тройка целей? (карточки с галочками / список / экран коллекции?)
- Показывать ли "сколько раз этот подарок был в твоих тройках"? (прогресс повторных целей)
- Нужно ли визуальное празднование закрытия тройки? (конфетти, звук, анимация — рекомендую да)

**Из Системы 7 (Авторизация):**
- Минимальная длина пароля? (рекомендую 4-6 символов для casual формата)
- Срок действия токена? (рекомендую бессрочный для упрощения)

**Общие вопросы презентации:**
- Тон feedback: игривый, ненавязчивый позитив (не восторженный, не нейтральный)
- Цветовая схема праздника Февромарта? (сочетание традиционных цветов 23 февраля + 8 марта?)
- Звуковое оформление: музыка фоновая? звуки действий? полная тишина?
- Мотивация дарить нужные подарки: как коллеги узнают что кому нужно? (показывать цели других? просить вживую? намекать через UI?)

### Для Level 7 (Parameters)

#### Формулы и Логика для Расчёта

| Система | Формула/Логика | Что настраивается |
|---------|----------------|-------------------|
| **Респавн предметов** | Каждые T минут добавляется N предметов на свободные точки | T = 5-10 мин (балансировка), N = 3-5 шт |
| **Трансформация подарка** | lookup_table[подарок_id] → новый_подарок_id | Генерация таблицы: правила (не в себя, разнообразие) |
| **Адаптация коллективной цели** | Цель = (подарено_сейчас / (время_прошло / 9ч)) × 0.9, bounds: [300, 1500] | Коэффициент 0.9 (насколько легко), bounds |
| **Генерация тройки целей** | random.sample(каталог, 3) без повторов в текущей тройке | Алгоритм выбора (чисто случайный / взвешенный по редкости) |

#### Параметры для Балансировки

| Параметр | Рекомендуемое значение | Зачем балансировать |
|----------|------------------------|---------------------|
| **Периодическая выдача: интервал** | ? минут | Слишком часто = нет мотивации идти на сцену, слишком редко = не помогает пассивным игрокам |
| **Респавн: интервал** | 5-10 минут | Слишком часто = переполнение сцены, слишком редко = нечего собирать |
| **Респавн: количество** | 3-5 предметов | Должно хватать на 20-30 игроков при частых заходах |
| **Точки на сцене** | 8-12 точек | Баланс визуальной плотности и вместимости |
| **Коллективная цель: минимум** | 300 подарков | Достижимо даже при 10 пассивных игроков |
| **Коллективная цель: максимум** | 1500 подарков | Не слишком гринди при 40 активных |
| **Коллективная цель: коэффициент** | 0.9 от прогноза | Баланс усилия и достижимости |
| **Пересчёт цели: время** | 12:00, 15:00, 17:00 | Достаточно часто для адаптации, не слишком часто (дёрганье цели) |
| **Длина сообщения** | 100-150 символов | Casual формат — короткие сообщения |
| **Минимальная длина пароля** | 4-6 символов | Баланс безопасности и удобства |
| **Размер каталога подарков** | 40-60 уникальных | При ~50 подарках на игрока за день, каждый встретится 1-2 раза |
