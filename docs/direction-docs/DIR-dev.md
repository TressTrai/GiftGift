# Direction Document: Разработчик

## Назначение роли

Разработчик отвечает за реализацию игровых систем, клиентской архитектуры, серверной логики, хранения состояния и технической связности проекта.

Проект остаётся небольшой одноразовой веб-игрой для корпоративного события, поэтому архитектура должна быть простой, проверяемой и устойчивой к типовым сбоям во время запуска.

## Архитектурный подход

- Фронтенд строится на Phaser 3, TypeScript и Vite.
- Бэкенд строится на Express и SQLite.
- Основной пользовательский режим — мобильный portrait, touch.
- Клиент и сервер разделяют ответственность: клиент знает содержание каталога подарков, сервер хранит состояние игры и оперирует идентификаторами.
- Состояние клиента централизовано в `GameStore`.
- Межсценовое взаимодействие идёт через `EventBus`; сцены не должны обращаться друг к другу напрямую.
- Все мутации игрового состояния проходят через методы store и эмитят события.
- Серверные изменения, связанные с общей игровой сценой, должны быть устойчивы к гонкам.

## Клиентская структура

Ключевые зоны:

- `src/main.ts` — конфигурация Phaser.Game и регистрация сцен.
- `src/types/index.ts` — общие типы клиента.
- `src/api/` — API-обёртки.
- `src/store/GameStore.ts` — singleton состояния игры.
- `src/systems/SyncSystem.ts` — polling, синхронизация и финальный таймер.
- `src/scenes/` — Phaser-сцены.
- `src/utils/constants.ts` — ключи сцен, события, цвета, точки респавна и числовые параметры.
- `src/utils/storage.ts` — localStorage для токена, туториала и звука.

## Архитектура сцен

Используется модель layered scenes:

- `GameScene` — основная интерактивная сцена.
- `UIScene` — постоянный оверлей с навигацией.
- `InventoryScene` и `ProfileScene` — вкладки поверх основной сцены.
- `ItemDetailScene`, `GiftingScene`, `RevealScene`, `FinaleScene` — модальные оверлеи.

При открытии инвентаря или профиля `GameScene` может быть переведена в sleep. UI-слой остаётся активным.

## Потоки данных

Вход в игру:

```text
BootScene -> catalog.json -> gameStore.initCatalog() -> fetchState() -> gameStore.init(state) -> GameScene + UIScene
```

Сбор предмета:

```text
Tap -> collectItem(instanceId) -> gameStore.applyItemCollected() -> EventBus.emit(ITEM_COLLECTED)
```

Дарение:

```text
ItemDetailScene -> GiftingScene -> sendGift(..., resultCatalogId) -> gameStore.applyGiftSent() -> EventBus.emit(GIFT_SENT)
```

Раскрытие:

```text
Tap wrapped gift -> RevealScene -> revealGift() -> gameStore.applyGiftRevealed() -> EventBus.emit(GIFT_REVEALED)
```

## Каталог подарков

`public/assets/catalog.json` существует только на клиенте.

Сервер не хранит названия, описания и `imageKey`. Он работает с `catalogId`, `instanceId` и состоянием владения.

При передаривании клиент выбирает `resultCatalogId`:

- item -> gift: сохраняется тот же `catalogId`;
- gift -> gift: выбирается случайный другой `gift-N`.

## Серверная логика

Бэкенд отвечает за:

- регистрацию и логин;
- хранение пользователей;
- инвентарь;
- нераскрытые подарки;
- предметы на сцене;
- личные тройки целей;
- коллективную цель;
- респавн предметов;
- проверку авторизации;
- транзакционную обработку сбора предметов.

Гонка при сборе предмета решается на сервере: первый успешный запрос получает предмет, следующий получает конфликт.

## Технические правила

- Не дублировать каталог подарков на сервере.
- Не связывать сцены прямыми ссылками.
- Не обходить `GameStore` при изменении клиентского состояния.
- Не показывать точное значение коллективной цели игрокам.
- Не добавлять лидерборды и соревновательные показатели.
- Сохранять мобильный portrait как основной формат интерфейса.
