# Level 4: Balance Paradigm

**Проект:** Февромарт — Игра Дарения Подарков  
**Дата:** 2 марта 2026  
**Depends on:** L0-context.md, L1-aesthetic.md, L2-dynamics.md, L3-temporal.md

---

## Paradigm Selection

- **Paradigm:** Hybrid — Adaptive для коллективной цели + Consistency для остальной механики

- **Rationale:**
  - **Adaptive для коллективной цели:** Защищает главный эмоциональный пик (19:00 финал). Gift-Giving Delight и Fellowship aesthetics требуют гарантированного shared triumph в конце дня. Без адаптации существует критический риск что коллективная цель не достигнется — это провалит весь эмоциональный arc игры и разрушит кульминацию.
  - **Consistency для остальной механики:** Всё остальное (респавн предметов, трансформация при дарении, анонимность) работает по фиксированным предсказуемым правилам. Это упрощает разработку (4 дня, 1 человек, 0₽ бюджет) и даёт игрокам ощущение контроля над своими действиями — они могут предсказать поведение системы и проявить игривость при выборе подарков.

- **What this commits us to:**
  - Адаптивная система мониторинга активности игроков и пересчёта коллективной цели в ключевые моменты дня
  - Скрытие механики адаптации от игроков — визуальный прогресс без точных цифр, магия сохраняется
  - Чёткие фиксированные правила для всех остальных систем — документация, тестирование, прозрачность для игроков
  - Баланс между "достаточно предсказуемо чтобы игроки могли стратегировать" и "главный пик защищён от провала"
  - Minimal scope для адаптации — только одна метрика, только одна цель адаптируется

## Implementation Details

### For Adaptive Paradigm (Коллективная Цель)

**Player state observed:**
- **Метрика:** Суммарное количество подарков которые игроки УЖЕ подарили друг другу к текущему моменту времени
- **Tracking mechanism:** Серверная логика считает каждое событие дарения:
  - Предмет → Подарок (первое дарение со сцены)
  - Подарок → Другой Подарок (передаривание)
- **НЕ отслеживается:** Личные коллекции, количество заходов, время в игре — только реальное дарение

**Response mechanisms:**
- **Пересчёт цели:** В три ключевых момента дня — 12:00, 15:00, 17:00
- **Логика пересчёта:**
  1. Система смотрит сколько подарков УЖЕ подарено к моменту пересчёта
  2. Экстраполирует темп: "К 19:00 будет примерно X подарков при текущем темпе"
  3. Устанавливает коллективную цель = X × 0.9 (чуть меньше прогноза — ощущение усилия, но достижимо)
  4. Между пересчётами цель фиксирована (не дёргается)
- **Визуализация:** Прогресс показывается визуально (например, "елка/пространство заполняется подарками"), БЕЗ точных цифр для игроков

**Adjustment bounds:**
- **Minimum goal:** 300 подарков — гарантия что даже при 10 очень пассивных игроках цель достижима за 9 часов
- **Maximum goal:** 1500 подарков — гарантия что при 40 гиперактивных игроках цель не станет слишком гринди или недостижимой
- **Rationale:** Bounds предотвращают edge cases (никто не зашёл / слишком много активности)

**Transparency:**
- **Скрытая адаптация:** Игроки НЕ знают что цель подстраивается под их активность
- **Что игроки видят:**
  - Визуальный прогресс заполнения (абстрактная шкала или дизайн пространства)
  - Возможно намёк на масштаб: "Сегодня нас X человек, украшаем пространство вместе!" — но без связи с конкретной числовой целью
  - Нет цифр типа "347/800 подарков"
- **Почему скрыто:** Сохраняет магию shared triumph — кажется что "мы вместе создали праздник", а не "система подстроила цифры под нас"

**Edge case handling:**

- **Цель достигнута раньше 19:00:**
  - Визуальный прогресс показывает 100%
  - Появляется праздничное сообщение: "Ура! Мы справились! Финальное празднование в 19:00!"
  - Игроки могут продолжать дарить подарки (прогресс остаётся на 100%, не переполняется)
  - Главная кульминация и финальное раскрытие — строго в 19:00 (сохраняет Crescendo к единому моменту для всех)
  
- **Цель не достигнута к 19:00 (маловероятно, но возможно):**
  - Adaptation bounds должны предотвратить это, но если произошло (баг / никто не играл):
  - Финал всё равно происходит в 19:00
  - Празднуется усилие: "Мы украсили пространство вместе! Спасибо всем!"
  - Не показывается провал — сохраняется позитивная эмоция

### For Consistency Paradigm (Фиксированные Правила)

**Rules are:** Фиксированные, детерминистические, предсказуемые для игроков

**1. Респавн Предметов в Сцене**

- **Механика:** Глобальный пул предметов для всех игроков (не персональная очередь)
- **Частота:** Каждые 5-10 минут в сцену добавляется 3-5 новых предметов
- **Размещение:** Предметы появляются на фиксированных точках в визуальной сцене (8-12 точек)
- **Доступность:** Предметы видны всем игрокам; кто быстрее собрал — тот получил (first-come-first-served)
- **Predictability:** Игроки понимают паттерн: "Если зайду через час, будут новые предметы"
- **Supporting изобилие:** Высокая частота респавна гарантирует что всегда есть что собрать даже при 30 активных игроках

**2. Трансформация При Дарении**

**Два типа трансформации:**

**A) Предмет → Подарок (первое дарение со сцены):**
- Игрок собрал предмет "Мишка" со сцены → у него в инвентаре предмет
- Предмет НЕ участвует в личных целях и общей цели (пока он у игрока)
- Игрок дарит предмет "Мишка" Лёше
- **Трансформация:** Предмет становится подарком (НЕ меняет вид — остаётся Мишкой)
- Лёша получает подарок "Мишка"
- Для Лёши это подарок — участвует в его личной коллекции и в общей цели

**B) Подарок → Другой Подарок (передаривание):**
- Игрок получил подарок "Мишка" от коллеги → у него в инвентаре подарок
- Игрок решает передарить "Мишку" Ване
- **В момент выбора дарения:**
  - Система детерминистически определяет трансформацию по ID этого экземпляра подарка
  - Например, этот конкретный Мишка (ID: 12345) → всегда трансформируется в Цветы
  - Показывается превью: "Ты даришь Мишку, Ваня получит: Цветы"
- **Игрок может:**
  - Подтвердить дарение → Ваня получит Цветы
  - Передумать и выбрать другой подарок из инвентаря
- **Determinism:** Один и тот же экземпляр подарка (по ID) всегда трансформируется в одно и то же
- **Разнообразие:** Если у игрока два "Мишки", у них разные ID → могут трансформироваться в разные подарки

**Почему это Consistency:**
- Правило предсказуемое: "предметы не меняются при первом дарении, подарки меняются при передаривании"
- Трансформация детерминистическая по ID (не случайная каждый раз)
- Игрок видит превью ДО дарения — может принять информированное решение ("Ване понравятся цветы? Да, подарю")

**3. Анонимность Дарения**

- **Механика:** При каждом дарении (предмет или подарок) игрок выбирает: "Анонимно?" (галочка/toggle)
- **Если анонимно:** Получатель видит "Тебе подарили [предмет/подарок]!" — БЕЗ имени дарителя
- **Если открыто:** Получатель видит "Лёша подарил тебе [предмет/подарок]!"
- **Свобода выбора:** Решение каждый раз — один подарок могу дарить анонимно, другой открыто
- **Supporting игривость:** Игрок может проявить внимание ("хочу чтобы Ваня знал что я подарил") или создать интригу ("интересно кто это?")

**4. Предотвращение Hoarding (Жадности)**

**Не через искусственные ограничения, а через естественную механику:**

- **Предметы изобильны:**
  - Постоянный респавн каждые 5-10 минут
  - Нет дефицита — не жалко дарить
  - Не участвуют в личных целях — смысл дарить, а не копить
  
- **Подарки ценны:**
  - Участвуют в личной коллекции (детали на Level 5)
  - Хочется оставить для коллекции
  - Создаётся естественный tension: "Этот оставлю себе, этот передарю коллеге"

- **Мотивация делиться:**
  - Визуальный прогресс коллективной цели постоянно виден → напоминает о вкладе
  - Социальное давление — видишь что другие дарят, хочется помочь (Fellowship)
  - Нет наказания за жадность, но есть позитивное притяжение к щедрости

- **Нет жёстких ограничений:** Инвентарь не ограничен искусственно — свобода игрока сохраняется

**Edge case handling:** Все правила работают одинаково в любой ситуации — нет исключений, нет "особых случаев"

## LLM Implications

Based on paradigm choice:

**LLM role permitted:** Pre-production content generation ONLY (не runtime интеграция)

**What LLM can do:**
- Генерация каталога подарков ДО релиза игры (40-60 уникальных подарков)
- Для каждого подарка:
  - Изображение (генерация через image generation API или сервис)
  - Короткое описание (1-2 предложения, игривое, праздничное)
- Генерация визуальной сцены для исследования (1 большая статическая иллюстрация праздничного пространства)
- Опционально: вариации подарков для разнообразия

**What LLM cannot do:**
- **Никакой runtime интеграции** — нет API вызовов во время игры (упрощает разработку, исключает latency, убирает затраты)
- **Нет adjudication** — все игровые правила механические (Consistency paradigm)
- **Нет interpretation** — адаптивная система использует простую арифметику (количество подарков), не требует LLM интерпретации player state
- **Нет dynamic content generation** — весь контент статический и подготовлен заранее

**Consistency requirements:**
- N/A для runtime (LLM не используется)
- Для pre-production генерации: стандартные настройки image generation models

**Latency tolerance:**
- N/A для runtime (LLM не используется)

**Failure handling:**
- N/A: LLM не используется в runtime, нечему ломаться во время игры
- Если контента сгенерировано недостаточно перед релизом — просто добавить ещё подарков в каталог вручную или через дополнительную генерацию

**Rationale:**
- **Hybrid paradigm не требует LLM в runtime:**
  - Adaptive часть отслеживает одну простую метрику (количество подарков) — это тривиальная арифметика
  - Consistency часть полностью детерминистическая — правила фиксированы
- **Ограничения проекта:**
  - 4 дня разработки + 1 человек + 0₽ бюджет → никакой runtime LLM интеграции (сложность, стоимость API, latency)
  - Статический контент можно сгенерировать заранее бесплатно или почти бесплатно
- **Wonder aesthetic:** Визуально приятные подарки критичны для эстетики — LLM генерация изображений экономит время художника
